# OBE AI 协作工作流 Demo 脚本

目标时长：10-15 分钟  
Demo 功能：合约交易页价格提醒提醒方式优化  
策略：优先实时执行；若 Lark/Figma/Codex 权限异常，切到预置结果，不影响分享主线。

## 0. 开场（1 分钟）

要点：

- 今天演示的不是“AI 替代 PM/设计师”，而是“把重复协作链路标准化”。
- 最终交付给开发的是 PRD + Figma/设计稿链接 + 验收标准/测试用例。
- Lark 是主入口，本地 Markdown 是源文件，Figma 是设计协作界面，Codex/Cursor 是执行器。

## 1. 输入一句需求（1 分钟）

输入：

```text
用户在合约交易页设置价格提醒时，希望能选择站内通知、邮件、Push 三种提醒方式，并能在触发前确认风险提示，避免错过关键价格或误以为系统会自动下单。
```

解释：

- 这不是一个真实未公开需求，只是沙盒功能。
- 它足够贴近业务：合约、提醒、风险提示、设计状态、测试验收都有。

## 2. 跑 feature-discovery（2 分钟）

提示词：

```text
召唤产品专家团队，对以下想法走 feature-discovery：
用户在合约交易页设置价格提醒时，希望能选择站内通知、邮件、Push 三种提醒方式，并能在触发前确认风险提示，避免错过关键价格或误以为系统会自动下单。
请输出做/不做/延后/缩小范围判断、目标用户、成功指标、非目标、关键风险、下一步入口。
```

预期输出：

- decision: narrow_scope
- 先做站内通知 + Push，邮件作为后续配置项
- 非目标：不做自动下单，不做复杂通知规则编排
- 下一步：进入 `prd-and-qa`

## 3. 跑 PRD + 验收（3 分钟）

提示词：

```text
基于需求澄清结论，走 prd-and-qa。
请输出 PRD 摘要、页面状态、核心规则、风险提示、验收标准和测试用例。
```

预期输出：

- PRD 摘要：价格提醒方式选择
- 页面状态：默认态、选择提醒方式、风险确认、创建成功、触发记录
- 验收标准：提醒方式可选、风险提示必读、不会暗示自动下单
- 测试用例：正常创建、未选方式、Push 权限关闭、触发提醒、重复提醒

## 4. 生成设计提示词（2 分钟）

提示词：

```text
基于 PRD，调用 web3-prd-figma-prompt 生成 Figma Make 设计提示词。
请包含页面结构、组件状态、风险提示、空态/异常态、移动端适配要求。
```

预期输出：

- 设计目标
- 页面结构
- 组件清单
- 状态清单
- 风险提示文案
- Figma 链接占位：`[待设计师补充 Figma 节点链接]`

## 5. 跑 plan-review-gauntlet（3 分钟）

提示词：

```text
召唤产品专家团队，对该 PRD 和设计提示词走 plan-review-gauntlet。
请分别输出产品战略、体验设计、工程计划、测试验收评审结论，并给出 pass / conditional_pass / fail。
```

预期输出：

- 产品战略：conditional_pass，建议先限制触达方式范围
- 体验设计：conditional_pass，需补 Push 权限关闭态
- 工程计划：conditional_pass，需确认通知服务能力和频控
- 测试验收：pass，测试边界清晰
- 总结：conditional_pass，可进入设计探索，不可进入开发排期

## 6. Lark 归档（2 分钟）

实时路径：

```bash
lark-cli docs +create --title "OBE AI 协作工作流 Demo - 合约价格提醒" --markdown @.codebuddy/docs/ai-product-workflow-sharing/demo-outputs/沙盒功能-Lark归档示例.md
```

回退路径：

- 直接展示本地 `demo-outputs/沙盒功能-Lark归档示例.md`
- 说明 Lark 登录或目标空间权限恢复后可用同一命令同步

## 7. 结束（1 分钟）

强调：

- PM 不是从空白页开始，而是从流水线开始。
- 设计师不是从含糊需求开始，而是从设计提示词和状态清单开始。
- 开发不是从聊天记录开始，而是从 PRD、设计稿、验收用例和联合评审结论开始。
