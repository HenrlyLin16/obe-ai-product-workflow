# Doc Writer

## 📌 描述
技术文档撰写专用 skill，负责加密货币交易所技术文档、产品文档、用户帮助文档的规范化撰写与维护，确保文档质量与一致性。

## 🎯 适用场景
- API 文档撰写
- 产品需求文档（PRD）编写
- 技术方案文档撰写
- 用户帮助文档编写
- 风控规则文档撰写
- 合约交易规则文档
- 运维手册编写
- 开发者文档维护

---

## 🔗 知识来源优先级

文档内容必须与项目既有规则和术语保持一致，本 Skill 在撰写/重构文档时统一遵循以下知识源优先级：

### 第一层：Notion 知识库（项目内部知识，优先级 0）

- **范围**：Web3 产品专用页面、项目 Wiki、文档数据库
- **内容**：
  - 历史 PRD、规则说明、技术方案、用户文档
  - 决策记录（ADR）、FAQ、风控/合约规则页
- **用途**：
  - 作为「**当前权威版本**」的唯一来源，避免新文档与历史结论冲突
  - 复用已有章节结构、术语与关键表述

### 第二层：NotebookLM（核心规则知识，优先级 1）

- **范围**：如《CEX 合约核心交易规则》等 NotebookLM 笔记本
- **内容**：永续合约核心规则、计算公式、风控逻辑、竞品对比等
- **用途**：
  - 当文档涉及保证金/强平/资金费等计算或合约规则时，为三段式公式和「规则说明」提供权威内容
  - 校验 PRD/规则文档中的公式与定义是否自洽

### 第三层：本地项目文档（优先级 2）

- **范围**：`/产品设计/`, `/合约/`, `/竞品调研/`, `/数据需求/`, `/多语言/` 等目录
- **内容**：
  - 现有本地 PRD、需求说明、技术设计、竞品调研
  - 术语对照表、多语言文案
- **用途**：
  - 对齐字段命名、接口签名、状态机与时序
  - 确保多语言术语一致

### OpenViking 本地上下文（可选，优先级 3）

- **范围**：已由 OpenViking 索引的 PRD/规则/文档目录
- **用途**：
  - 需要从大量历史文档中**快速抽取片段**时，用语义检索获取上下文，再整理为正式文档

### 文档撰写查询流程

```text
撰写/更新文档（PRD / 规则 / 技术 / 帮助）
    ↓
1. 查询 Notion（历史版本 / 相关决策 / 术语）
    ↓ 找到
    ← 在此基础上增量修改或重构，保持结论一致
    ↓ 未找到或不完整
2. 查询 NotebookLM（核心规则、公式、定义）
    ↓ 找到
    ← 用于填充「规则说明」「计算示例」等部分
    ↓ 未找到
3. 查询本地项目文档（产品设计/合约/竞品调研/多语言）
    ↓ 找到
    ← 对齐字段、状态机、接口与文案
    ↓ 仍未找到
4. 结合行业通用实践起草草案，并显式标注「来源：行业实践；上线前需在 Notion 固化为权威版本」
```

## 📋 使用方法

**调用方式**：在对话中使用 `skill: doc-writer` 指令

**输入参数**：
- `doc_type`: 文档类型（API/产品/技术/帮助/风控）
- `target_audience`: 目标读者（开发者/用户/产品经理/运营）
- `language`: 语言（简体中文/繁体中文/英文）
- `feature`: 功能名称
- `references`: 参考文档路径

**输出格式**：
1. **API 文档**：接口定义、参数说明、示例代码
2. **产品文档**：PRD、功能说明、使用指南
3. **技术文档**：架构设计、算法逻辑、技术方案
4. **帮助文档**：教程、FAQ、故障排除
5. **风控文档**：规则说明、计算公式、示例

## 🎨 输出模板

### API 文档模板
```markdown
# [API 名称] 接口文档

## 1. 接口概述
- 接口名称：[名称]
- 接口路径：`[GET/POST/PUT/DELETE] /api/v1/path`
- 功能描述：[一句话描述]
- 认证方式：[API Key/签名/JWT]

## 2. 请求参数

### 2.1 路径参数
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| param1 | string | 是 | [说明] |

### 2.2 查询参数
| 参数名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| symbol | string | 是 | 交易对 | BTCUSDT |

### 2.3 请求体
```json
{
  "key1": "value1",
  "key2": 123
}
```

## 3. 响应参数

### 3.1 成功响应
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "field1": "value1"
  }
}
```

### 3.2 错误响应
| 错误码 | 说明 | 解决方案 |
|--------|------|----------|
| 40001 | 参数错误 | 检查参数格式 |
| 40101 | 签名错误 | 检查签名算法 |

## 4. 请求示例

### 4.1 cURL 示例
```bash
curl -X GET "https://api.example.com/v1/path?symbol=BTCUSDT" \
  -H "Authorization: Bearer YOUR_TOKEN"
```

### 4.2 Python 示例
```python
import requests

url = "https://api.example.com/v1/path"
headers = {"Authorization": "Bearer YOUR_TOKEN"}
params = {"symbol": "BTCUSDT"}

response = requests.get(url, headers=headers, params=params)
print(response.json())
```

## 5. 限流规则
- 速率限制：[X 次/秒]
- 权重限制：[Y 权重/请求]

## 6. 注意事项
- [重要提示 1]
- [重要提示 2]
```

## 🔍 文档规范

### 1. 术语一致性
严格遵循多语言术语库，确保术语统一：
- 参考 `/多语言/` 目录下的术语对照表
- 同一术语在所有文档中使用相同翻译
- 首次使用时提供中英文对照

### 2. 格式规范
- 标题层级：使用 `#` `##` `###` 最多不超过 3 级
- 代码块：使用 ``` 语言名 格式
- 表格：使用 Markdown 表格格式
- 列表：使用有序/无序列表
- 强调：使用 `**粗体**` 或 `*斜体*`

### 3. 编写原则
- **清晰**：避免模糊表述
- **准确**：数据、公式、示例必须准确
- **简洁**：去除冗余信息
- **结构化**：使用标题、列表、表格组织内容
- **示例化**：提供具体示例帮助理解

### 4. 多语言支持
- 优先编写简体中文版本
- 根据需求生成繁体中文、英文版本
- 确保术语在不同语言中的一致性

### 5. 文档评审
- 技术文档需经技术评审
- 产品文档需经产品评审
- 用户文档需经用户体验评审
- 风控文档需经风控评审
