# 需求文档生成 Figma Make 提示词

## 📌 描述
本 SKILL 专用于 Web3 产品专家在完成 PRD 撰写后，自动生成 Figma Make 界面设计的专用提示词，将产品需求文档转化为可直接用于 Figma Make UI 设计的结构化设计说明。

## 🎯 适用场景
- 在 PRD 文档撰写完成后，需要生成 Figma 界面设计提示词
- 基于 OneBullEX 交易界面的现有风格，设计新功能界面
- 为设计师提供清晰的设计需求和约束条件
- 确保设计与产品需求的一致性
- 保持 OneBullEX 界面设计风格的统一性

---

## 🔗 知识来源优先级

本 Skill 将「PRD → 设计提示词」桥接起来，必须首先尊重项目内部的**权威产品与设计规范**。

### 第一层：Notion 知识库（产品与设计源，优先级 0）

- **范围**：Web3 产品专用页面、设计规范页面、已归档的 PRD
- **内容**：
  - 最新版 PRD 与产品规则
  - 设计系统/组件库/设计规范（如有）
  - 历史上相似功能的设计结论与评审意见
- **用途**：
  - 作为生成提示词时「**产品侧单一事实来源**」
  - 保证提示词中功能范围、约束条件与正式 PRD 保持一致

### 第二层：NotebookLM（规则背景，优先级 1）

- **范围**：规则与合约相关的 NotebookLM 笔记本
- **用途**：
  - 当 PRD 中对规则部分描述较简略，而提示词需补充「风险提示/数据字段含义」时，可从规则文档中获取统一表述

### 第三层：本地项目文档（优先级 2）

- **范围**：`/产品设计/`, `/用户体验/`, `/竞品调研/` 等目录
- **内容**：
  - 本地 PRD、交互设计文档、设计规范文件
  - **`产品设计/设计提示词/OBE DESIGN.md`**（OneBullEX 配色 / 字体 / 按钮与 OTP 规范快照、Agent Prompt 摘录、CSS Token；生成 Figma Make 提示词时 **必须引用** 并与 PRD 范围对齐）
  - 竞品界面分析报告（用于给出参考风格）
- **用途**：
  - 对齐字段命名、布局模式、交互流程描述

### OpenViking 本地上下文（可选，优先级 3）

- **范围**：包含 PRD 与设计文档的已索引目录
- **用途**：
  - 当用户未显式给出 PRD 路径，仅给主题时，先基于语义检索找到对应 PRD/设计方案，再生成提示词

### 提示词生成前的查询流程

```text
PRD 完成，需要生成 Figma Make 提示词
    ↓
1. 从 Notion 或本地目录获取「最终版 PRD」与相关设计规范
    ↓
2. 如涉及合约/风控/资金等规则敏感区块，可查 NotebookLM/规则文档，统一措辞与字段含义
    ↓
3. 由本 Skill 将 PRD 内容结构化为提示词骨架，严格不扩展超出 PRD 范围的功能
    ↓
4. 对于需要设计师创作空间的部分，使用占位标记并在提示词中说明
```

## 🔧 核心能力

### 1. PRD 到设计提示词的转化
- 自动分析 PRD 文档的核心功能和交互需求
- 提取关键界面元素和交互流程
- 生成 Figma Make 可理解的界面设计描述
- 确保设计提示词符合 Figma Make 的输入格式要求

### 2. OneBullEX 设计风格适配
- 保持与 OneBullEX 现有界面的一致性
- 遵循 OneBullEX 的设计规范（配色、字体、间距、圆角等）
- 参考 OneBullEX 交易界面的布局模式
- 确保 Web3/加密货币交易所的行业标准视觉元素
- **图标**：线框/描边默认 **[Feather Icons](https://github.com/feathericons/feather)**（名称与 [feathericons.com](https://feathericons.com) 一致；取用见 [web3-product-expert.md §1.1b](web3-product-expert.md)）。原型阶段若设计师从 **中文检索** 或 **SVG 粘贴** 更快，可注明允许 **[iconfont](https://www.iconfont.cn/)**（流程与授权见 **§1.1c**）

### 3. 功能模块设计
- K 线图表显示设置面板设计
- 快捷操作浮层设计
- 订单管理界面设计
- 交易下单界面优化
- 风险提示和确认弹窗设计

### 4. 交互设计说明
- 清晰描述交互流程和用户操作路径
- 说明 hover 状态、active 状态、disabled 状态
- 明确弹窗、下拉菜单、浮层的触发方式
- 定义键盘快捷键操作方式

## 📋 使用方法

### 触发时机
在 Web3 产品专家完成 PRD 文档撰写并得到用户确认后，自动触发以下确认流程：

```
用户确认 PRD 完成 → AI 提示："是否基于已生成的 PRD 文档，自动生成 Figma Make 界面设计的专用提示词？"
```

### 用户响应
- **用户选择「是」**：自动生成 Figma Make 设计提示词
- **用户选择「否」**：跳过此步骤，继续后续工作

## 🔄 工作流程

### 阶段 1：PRD 分析
- 读取已生成的 PRD 文档
- 提取核心功能需求
- 识别关键界面元素和交互流程
- 确定设计优先级

### 阶段 2：设计提示词生成
- 基于 PRD 内容，生成结构化 Figma Make 提示词
- 自动填充界面布局、组件设计、交互设计等章节
- 引用 OneBullEX 现有设计规范
- 标注需要设计师补充的占位内容

### 阶段 3：输出与确认
- 输出完整的 Figma Make 设计提示词
- 询问用户是否需要调整或补充
- 确认无误后，保存设计提示词文档

### 阶段 4：质量检查
- 检查设计提示词是否覆盖所有 PRD 功能需求
- 检查是否符合 OneBullEX 设计风格
- 检查是否遗漏关键交互场景
- 检查设计规范的完整性和一致性

## 📊 设计提示词生成规则

### 自动填充规则
- **功能名称**：自动从 PRD 标题提取
- **界面类型**：根据功能自动判断（面板/弹窗/浮层）
- **核心功能**：从 PRD 的「产品功能详细说明」章节提取
- **交互流程**：从 PRD 的「正常流程」章节提取
- **边界场景**：从 PRD 的「边界场景」章节提取

### OneBullEX 设计规范引用
- 自动引用 OneBullEX 标准配色体系
- 自动引用 OneBullEX 字体规范
- 自动引用 OneBullEX 间距和圆角规范
- 自动标注需要设计师补充的品牌元素

### 占位内容处理
- 需要设计师补充的内容使用 `[占位内容]` 标记
- 需要确认的设计决策使用 `[待确认]` 标记
- 需要多方案对比的内容使用 `[方案 A / 方案 B]` 标记

## ⚙️ 配置参数

- 默认保存路径：`产品设计/设计提示词/`

### 输出格式
- Markdown 格式

## ⚠️ 注意事项

1. **一致性优先**：确保设计提示词与 OneBullEX 现有界面风格一致
2. **PRD 同步**：设计提示词必须严格基于已确认的 PRD 内容
3. **可落地性**：生成的提示词必须能够被 Figma Make 理解并执行
4. **完整性**：覆盖所有 PRD 中提到的功能和交互场景
5. **灵活性**：为设计师留出创意空间，不要过度限制设计决策
6. **可追溯性**：标注每个设计要求对应的 PRD 章节
