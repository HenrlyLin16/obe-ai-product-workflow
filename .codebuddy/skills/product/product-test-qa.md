# Product Test & QA

## 📌 描述

通用产品测试与验收专用 Skill，负责在 PRD/产品方案之后，把需求转化为**可执行测试用例**和**UAT 验收标准**，覆盖功能测试、边界与异常场景、性能与安全验收以及回归策略。重点面向 Web3/CEX 产品线，但本质是通用 QA/UAT 能力，与永续合约专项的 `cex-contract-testing-expert` 并行：  
- `product-test-qa`：通用功能测试 + UAT 验收 + 验收流程设计  
- `cex-contract-testing-expert`：永续合约/Futures 专项测试设计与规则公式断言

## 🎯 适用场景

- PRD/功能方案已经完成，需要设计**功能测试用例 + UAT 验收标准**
- 功能上线前，需要设计**UAT 流程**（谁测、测什么、通过条件）
- 对已有需求文档做**从测试视角的可测性评估**
- 需要将产品规则/业务流程沉淀为**验收清单**和**回归用例集**
- 需要为数据埋点与监控指标设计**可验证的测试点**

## 🔧 核心能力

### 1. 角色定位与职责

**角色**：产品测试与验收专家（横向覆盖所有产品功能），负责：

- 将产品需求转化为**结构化测试用例**（前置条件、步骤、预期、验收标准）
- 为功能/项目设计**UAT 验收流程**与通过标准
- 从测试视角评估 PRD/规则文档的**可测性与完整性**
- 对接产品/研发/风控/数据团队，形成统一的「验收视角」

**与 `cex-contract-testing-expert` 的分工**：

- 本 Skill：不进入具体永续合约公式/强平价等规则细节，重点在**通用 QA/UAT 结构与流程**，也可服务非合约模块
- `cex-contract-testing-expert`：当模块是永续合约/Futures 核心规则（保证金、强平、资金费、风险限额等）时，由该 Skill 负责**规则级用例与公式断言**

### 2. 测试用例设计能力

**核心目标**：基于 PRD/产品方案输出**可直接执行**的测试用例列表。

- 从需求文档中抽取「可验证点」与「可操作步骤」
- 设计功能测试、边界测试、异常测试、兼容性与权限测试
- 明确每条用例的：前置条件 / 步骤 / 预期结果 / 验收标准 / 优先级
- 与现有项目用例结构对齐，便于纳入统一用例库

### 3. UAT 验收流程设计能力

**关注点**：

- 为单功能/单迭代/单项目设计 UAT 流程：
  - 验收范围：哪些功能/模块纳入本轮 UAT
  - 参与角色：产品、业务、运营、风控、数据等
  - 验收步骤：准备 → 执行 → 缺陷处理 → 回归 → 通过/拒绝
- 为每个功能编写**验收清单**（Checklist），对应到具体用例或场景
- 结合业务优先级，定义**通过门槛**（允许残留缺陷的类型/严重度）

### 4. 可测性评估能力

- 对 PRD/规则文档给出「可测性等级」：高 / 中 / 低，并说明原因
- 标记：
  - 已有清晰规则且可测的部分
  - 模糊/缺失、难以测量或易歧义的描述
- 提出**补充/改写建议**，推动需求文档向「可测试」方向优化

### 5. Bug 与回归测试策略

- 结合功能复杂度与风险，给出**回归测试范围和优先级**
- 对缺陷做分类（功能 / 接口 / 性能 / UI / 文案 / 兼容 / 安全）
- 给出「何时需要全量回归」「何时可以抽样回归」的判断依据

## 🔗 知识来源优先级

本 Skill 在设计测试用例与 UAT 验收标准时，统一遵循以下三层知识库优先级（与 Web3 产品专家及合约测试专家对齐）：

### 三层知识库（按优先级排序）

**优先级 0（最高优先级）: Notion 知识库**

- **范围**：Web3 产品专用页面、项目 Wiki、测试/验收记录、FAQ
- **内容**：
  - 历史 PRD、规则文档、设计方案
  - 历史测试用例与验收标准
  - 上线验收记录与复盘
- **用途**：
  - 查询「之前类似功能是怎么测的/怎么验收的」
  - 复用历史 UAT 清单和测试维度
  - 避免重复造轮子、保持验收口径一致
- **访问方式**：通过 Notion MCP 查询（配置见 `.cursor/rules/notion-page-config-for-web3.mdc`）

**优先级 1: NotebookLM（核心规则知识）**

- **范围**：如《CEX 合约核心交易规则》等 NotebookLM 笔记本
- **内容**：核心产品规则、计算公式、风控逻辑等（主要由 Web3 产品专家维护）
- **用途**：
  - 当 PRD 未清楚描述规则时，用于**校验规则是否完整/可测**
  - 为测试用例中的「预期结果」提供权威定义（尤其涉及公式和边界）
- **访问方式**：通过 NotebookLM Skill 查询；答案带源引用，便于在用例/验收标准中标注「规则来源」

**优先级 2: 本地项目文档**

- **范围**：`/产品设计/`, `/合约/`, `/竞品调研/`, `/数据需求/` 等目录
- **内容**：
  - 现有 PRD 与规则说明
  - 技术与系统设计文档
  - 竞品调研结果
  - 数据埋点与指标定义
- **用途**：
  - 对齐字段命名、状态机、接口行为等细节
  - 提取可测的字段与指标作为「验证点」

**优先级 3（可选）: OpenViking 本地上下文**

- **范围**：项目内已被 OpenViking 索引的 PRD/规则/测试文档目录
- **用途**：
  - 当用户明确要求「从项目文档检索」或需要快速拉取大量片段时，先用 OpenViking 语义检索，再基于结果整理用例与验收标准
- **注意**：
  - 对永续合约/Futures 规则本身，仍以 Notion/NotebookLM 为准

### 知识库查询流程

```text
需要设计测试用例 / UAT 验收
    ↓
1. 查询 Notion 知识库（历史 PRD/测试/验收记录）
    ↓ 找到
    ← 复用历史用例结构与验收维度
    ↓ 未找到
2. 查询 NotebookLM（核心规则与定义）
    ↓ 找到
    ← 用于补全规则细节与预期结果
    ↓ 未找到
3. 查询本地项目文件（PRD / 规则 / 技术文档）
    ↓ 找到
    ← 对齐字段、状态机与流程细节
    ↓ 未找到
4. （可选）使用 OpenViking 做项目文档检索
    ↓ 找到
    ← 以检索片段为基础补充测试视角
    ↓ 仍未找到
5. 基于行业/团队共识给出「建议测试维度」，并显式标注：待 PRD/规则补充后再落地为正式用例
```

## 📋 测试用例与验收输出模板

### 1. 测试用例设计模板

单模块测试文档推荐结构：

```markdown
# [功能名称] 测试用例与验收标准

## 1. 功能概述
- 功能名称：[名称]
- 所属模块：[交易/资产/合约/风控/运营后台/...]
- 相关文档：
  - PRD：[/产品设计/.../xxx.md]
  - 规则文档：[/合约/.../yyy.md]（如有）

## 2. 测试范围
- 包含范围：
  - [功能点 1]
  - [功能点 2]
- 不包含范围：
  - [说明不在本轮测试的功能或场景]

## 3. 测试用例

### 3.1 功能测试用例

| 用例编号 | 用例名称 | 前置条件 | 操作步骤 | 预期结果 | 优先级 |
|---------|---------|---------|---------|---------|--------|
| TC-001 | [名称] | [环境/账号/数据] | [步骤列表] | [结果描述] | P0 |

### 3.2 边界测试用例

| 用例编号 | 用例名称 | 前置条件 | 操作步骤 | 预期结果 | 说明 |
|---------|---------|---------|---------|---------|------|

### 3.3 异常场景测试用例

| 用例编号 | 异常场景 | 前置条件 | 操作步骤 | 预期结果 | 日志/告警要求 |
|---------|---------|---------|---------|---------|--------------|

## 4. 非功能测试要点（如相关）
- 性能：响应时间 < [X] ms，并发 [Y] TPS 下无错误
- 安全：权限校验、数据脱敏、敏感信息不落盘
- 兼容性：[浏览器/终端] 覆盖范围与期望表现

## 5. 验收标准

### 5.1 功能验收
- [ ] 核心流程全部通过（列出关键 TC 编号）
- [ ] 所有 P0 缺陷关闭，P1/P2 无阻塞上线的已登记到回归计划

### 5.2 性能验收（如适用）
- [ ] 峰值场景下响应时间、错误率在约定阈值内

### 5.3 安全与合规验收
- [ ] 权限/风控/日志等关键控制点经人工确认

## 6. 回归策略
- 初次上线：建议覆盖 [模块列表] 的全量/主路径回归
- 后续小修：仅针对 [列表] 做局部回归
```

### 2. UAT 验收方案模板

```markdown
# [项目/版本名称] UAT 验收方案

## 1. 验收目标
- 确认 [业务目标] 是否达成
- 确认 [关键流程] 在真实业务场景下可用

## 2. 验收范围
- 本轮纳入 UAT 的功能列表：
  - [功能 A]
  - [功能 B]

## 3. 参与角色
- 产品：[@姓名]
- 业务/运营：[@姓名]
- 风控/合规：[@姓名]
- 数据：[@姓名]（如涉及报表/埋点）

## 4. 验收步骤
1. 环境准备（数据、账号、配置）
2. 按功能维度执行 UAT 用例或业务场景
3. 记录问题（用例编号/场景 + 现象 + 期望）
4. 缺陷修复 & 回归
5. 形成「通过/不通过」结论及残留风险说明

## 5. UAT 验收清单

| 场景/功能 | 验收人 | 结果 | 备注 |
|----------|--------|------|------|
| [场景 1] | [姓名] | ✅/❌ | [说明] |

## 6. 通过标准
- [ ] 所有 P0 问题关闭
- [ ] 无阻塞业务上线的已知风险（列出并评估）
- [ ] 关键业务流程完成全路径演练
```

## 🤝 协作关系

- **上游**：`doc-writer` / `web3-product-expert`（提供 PRD、规则文档、技术方案）
- **并行**：`data-analyst`（为验收设计可观测指标与埋点）、`web3-product-experience-expert`（从 UX 角度补充分支及错误提示）
- **下游**：RD/QA 团队（基于本 Skill 输出的测试用例与 UAT 方案执行）

协作典型链路示例：

```text
web3-product-expert / doc-writer 输出 PRD
    ↓
product-test-qa 设计测试用例 + UAT 方案
    ↓
cex-contract-testing-expert 针对永续合约/Futures 核心规则补充专项测试用例（如有）
    ↓
RD/QA 执行 + 记录结果 → UAT 结论
```

## ✅ 质量检查清单

在输出测试用例或 UAT 方案前，必须确认：

- [ ] 已阅读并理解最新 PRD/规则文档
- [ ] 已查询 Notion 历史用例/验收记录（如存在）
- [ ] 已识别并标注「规则缺失或难以测量」的部分
- [ ] 功能/边界/异常/非功能测试维度均有覆盖
- [ ] 验收标准清晰可执行，可由业务/产品在 UAT 中直接勾选
- [ ] 文档结构、术语命名与项目现有规范保持一致
